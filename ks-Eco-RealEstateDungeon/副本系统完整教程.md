## 1. 前置条件

| 条件 | 说明 |
|------|------|
| 服务器已重启 | extra JAR 改动后必须整服 restart（plugman reload 不重置 extra classloader）|
| `ks-InstanceWorld` 已启用 | 本模块通过其服务申请实例、粘贴地图并释放网格；服务不可用时不能开本 |

| 玩家在线 | 开本会把全队传送进虚空世界；离线成员不计入开本人数 |
| 有 admin token | 建模板 / 改配置 / 看网格 / 强制结束需要管理员 |

## 2.【管理员·网页】建副本模板 + 配置

后台「🏰 副本系统 → 副本模板」填表保存即可（留空 ID 自动生成）。运行时模板字段包括：名称 / 难度 /
门票价 / 最少人数 / 最多人数 / 时限(分) / 怪物等级 / 描述 / Schematic / 房产钥匙要求 / `reward_config`。

> **最少/最多人数**就是玩家开本时的组队人数校验区间（见第 4 节）。

「系统配置」可热更新（标「即时」立即生效，标「下次副本」仅对之后新副本生效）：

| key | 默认 | 生效 | 含义 |
|-----|------|------|------|
| `ticket.default_price` | 500 | 即时 | 默认门票价 |
| `revive.base_cost` | 200 | 即时 | 第 1 次复活费 |
| `revive.exponent` | 1.8 | 即时 | 复活费指数 `cost=base×exp^(n-1)` |
| `revive.max_revives_per_player` | 10 | 即时 | 每人复活上限 |
| `grid.world_name` | ks-dungeon-world | 下次副本 | 虚空世界名 |
| `grid.spacing` | 5000 | 下次副本 | 网格间距 |
| `grid.max_grids` | 64 | 下次副本 | 最大并发网格 |
| `grid.instance_timeout_minutes` | 60 | 即时 | 副本超时 |
| `map.base_y` | 64 | 下次副本 | Schematic 粘贴基准高度 |
| `map.arena_radius` | 256 | 下次副本 | 实例清理的最小水平范围；实际还会合并 Schematic 边界 |
| `map.prepare_timeout_seconds` | 120 | 下次副本 | 世界准备、粘贴和标记扫描总超时 |
| `map.spawn_mobs` | true | 即时 | 是否解释怪物标记并调用 MythicMobs |

> 房产相关配置（开发费/契税/容积率）已迁到 ks-Eco-RealEstate，不在这里。

curl 等价（建模板）：
```bash
curl -s -X POST "http://localhost:8123/ks-Eco/api/realestate-dungeon/templates" \
  -H "Authorization: Bearer $ADMIN" -H "Content-Type: application/json" \
  -d '{"id":"raid-normal","name":"普通副本","ticketPrice":500,"minPlayers":1,"maxPlayers":4,"timeLimitMinutes":60}'
```

## 4.【玩家·指令】组队 → 开本 ★核心★

玩家全程用 `/dungeon` 指令（先组队、再开本，开本时校验人数）：

| 指令 | 作用 |
|------|------|
| `/dungeon` | 面板：可用模板、你的队伍、当前副本、剩余网格 |
| `/dungeon invite <玩家>` | 队长邀请玩家（2 分钟内有效）|
| `/dungeon accept` | 接受邀请，加入队伍 |
| `/dungeon party` | 查看当前队伍 |
| `/dungeon start <模板ID>` | **队长开本**：校验在线队伍人数 ∈ [最少,最多]，队长付一张门票，全队一起传入 |
| `/dungeon leave` | 离开副本 / 退出队伍 |
| `/dungeon revive` | 在副本内付费复活 |

**典型流程（3 人副本）：**
```
队长> /dungeon invite Alice
Alice> /dungeon accept
队长> /dungeon invite Bob
Bob> /dungeon accept
队长> /dungeon party          # 确认 3 人
队长> /dungeon start raid-normal
```
- 人数 < 最少 → `人数不足：该副本需至少 N 人`
- 人数 > 最多 → `人数超限：该副本最多 N 人`
- 单人开本：不组队直接 `/dungeon start <模板>`，要求该模板 minPlayers ≤ 1。

**开本背后发生了什么（虚空副本怎么生成）：**
1. 冻结本次队伍名单，在数据库写入门票 `CHARGE_READY→CHARGE_IN_PROGRESS`。
2. 回到服务器线程由队长通过 Vault 付款；成功后把门票、`WAITING` 实例和参与者一起提交。
3. 通过 `ks-InstanceWorld` 申请网格、清场、粘贴模板 Schematic，并分批扫描 `[spawn]` / 怪物标记。
4. 准备成功后提交 `ADMITTED` 与实例 `WAITING→ACTIVE`，解释怪物标记并传送在线成员。
5. 任一步骤失败都会释放实例句柄；已扣款但未能入场的路径进入持久化退款流程，而不是只靠内存回调。

> Schematic 现在是正常开本的必需模板字段。副本内容与遭遇仍由地图、MythicMobs 等外部内容提供；
> 本模块不在 Java 中硬编码某个 Boss 或场景。

### 4.1 门票结算账本

`ks_dungeon_ticket_settlements` 记录每个实例唯一的一次门票结算：

`CHARGE_READY → CHARGE_IN_PROGRESS → CHARGED → ADMITTED`

失败/恢复分支使用 `REFUND_READY → REFUND_IN_PROGRESS → REFUNDED`，以及 `REJECTED`、`CANCELLED`、
`REVIEW_REQUIRED`。插件启动时会恢复本节点遗留的可退款记录；如果停机发生在外部 Vault 扣款或退款调用中，
结果无法由数据库单独证明，记录会停在 `REVIEW_REQUIRED`，必须先对账，不能盲目重扣或重退。

## 5. 副本内房产（已迁至房地产插件）

副本内买地/开发现在由 **ks-Eco-RealEstate** 提供（共用 `ks_re_plots`，副本地块 `instance_id` 非空）。Web API（玩家 token）：
```bash
# 在副本内买地
curl -s -X POST "http://localhost:8123/ks-Eco/api/realestate/instance/plot/purchase" \
  -H "Authorization: Bearer $PLAYER" -H "Content-Type: application/json" \
  -d '{"instanceId":"DI1a2b3c4d","x1":0,"z1":0,"x2":16,"z2":16,"propertyFunction":"DUNGEON_PORTAL","price":1000}'
# 缴开发配套费
curl -s -X POST ".../api/realestate/instance/plot/develop" -H "Authorization: Bearer $PLAYER" -H "Content-Type: application/json" -d '{"plotId":"<id>"}'
# 我的副本房产
curl -s ".../api/realestate/my-instance-plots?instanceId=__any_dungeon__&token=$PLAYER"
```
副本销毁时，其 `instance_id` 对应的地块会被一并清除。

## 6. 死亡与复活（硬核指数费）

副本内死亡保留背包/等级、记为 DEAD。复活：游戏内 `/dungeon revive`，或 API：
```bash
curl -s -X POST ".../api/realestate-dungeon/revive" -H "Authorization: Bearer $PLAYER" \
  -H "Content-Type: application/json" -d '{"instanceId":"DI1a2b3c4d"}'
```
费用 `cost=base×exp^(n-1)`（默认 200/1.8）：第 1/2/3/5/10 次 ≈ 200/360/648/2098.8/4640.45；超 `max_revives_per_player` 拒绝。

## 7. 生命周期与清理

```
WAITING ─(实例准备并提交入场)→ ACTIVE ─┬─(全部登记 Boss 死亡)→ COMPLETED
                                      ├─(超时 expiresAt)→ ABANDONED
                                      └─(admin force-end / 全员 LEFT)→ ABANDONED

参与者：ALIVE ↔ DEAD；主动离开或终态清理 → LEFT
奖励：实例 reward_status NONE → PENDING → GRANTED
结束后：删除实例房产 → 请求 ks-InstanceWorld 幂等释放 → 清场并把网格归还 FREE
```

`LEFT` 是持久化资格状态，不等同于暂时掉线。玩家在通关快照生成前执行 `/dungeon leave` 后不会进入奖励名单；
仍在副本名单中的 `ALIVE`/`DEAD` 成员会被写入 `ks_dungeon_reward_roster`，随后所有剩余参与者才统一转为 `LEFT`。

### 7.1 多 Boss 通关判定

- 兼容标记：`[mm]/怪物名:等级/boss`。
- 通用标记：`[marker]/mm/怪物名:等级/boss`。
- 名称含 `controller` / `level_controller` 的 MythicMobs 控制器也会作为完成目标。
- 同一实例会保存所有成功生成的完成目标 UUID；监控每约 5 秒检查一次，只有集合中全部目标均不再存活才通关。
- 没有登记任何完成目标的实例不会自动通关；目标生成失败会写警告日志。

当前完成集合是本次运行期状态，并不提供崩溃后的战斗续接。正常插件停用会把未结束实例标记为放弃并释放；
内容包里的 `completion-boss-id` 不是当前模板字段，实际完成条件以地图标记生成出的 UUID 集合为准。

### 7.2 奖励账本与 proof 状态

模板 `reward_config` 可声明金钱、控制台命令、MythicMobs 物品和 MMOItems。副本完成后，系统先冻结奖励名单，
再为每个“玩家 + 奖励项”创建 `ks_dungeon_reward_grants` 记录：

- `NONE`：尚未领取；
- `PENDING`：本次发放已认领，或外部结果需要人工确认；
- `RETRY_REQUIRED`：明确失败，可在自动重试上限内再次认领；
- `GRANTED`：该奖励项已确认完成。

只有全部逐项记录都为 `GRANTED`，实例 `reward_status` 才会从 `PENDING` 进入 `GRANTED`。普通概率未命中会作为
已决的 `SKIPPED` 完成，不会在重启后重新抽取。

对于精确形态的 `ksrpg proof grant <玩家> <proofId>` 命令，奖励桥会在命令返回后通过
`RpgProgressionApi.hasProof` 再检查一次。目标离线、ks-RPG 服务不可用、命令结果未知或 proof 未实际写入时，
记录不会伪装成成功：它会保持 `PENDING` 待核对，或进入 `RETRY_REQUIRED`。proof 奖励应配置 `chance: 1.0`；
proof 的概率未命中也会被视为需要人工核对。

仍未接线的部分：副本尚未直接调用一个支持离线、幂等恢复的 proof 结算 API；内容包中的 `entry-gate` 仍是设计
元数据，运行时不会据此拒绝入场；材料、`CASH` 与 proof 的原子突破事务也尚未实现。因此这些 gate/突破配置在
完成实机闭环前不能作为正式生产门槛。

## 8. 与 ks-RPG 内容和赛季基础的边界

- ks-RPG 启动时会从 JAR 提取缺失的 `weapons`、`talismans`、`rings`、`caches`、`world-drops` YAML，
  使用“不覆盖”模式保留服务器已有文件。
- `/ksrpg reload` 先加载候选 `config.yml` 和上述五类已识别内容；参与加载的 YAML、proof gate、机制名、
  概率/权重等只要有一项无效，就保留上一份完整运行目录，不做半套替换。
- 配置掉落记录对目标造成有效伤害的玩家/玩家投射物；死亡时要求玩家击杀，并在同世界、在线、非旁观且距目标
  64 格内的合格参与者之间按掉落表轮转已掷出的物品，不再固定全部给最后一击。
- `season.enabled` 默认是 `false`。即使手工开启，当前也只会在专用工作线程初始化本地
  `plugins/ks-RPG/season.db` 和只读状态服务；不会自动创建赛季、调度事件、发奖励或修改玩家数据。

副本奖励仅消费 ks-RPG 已公开的 proof/物品能力，不负责启动赛季。跨服 proof/season 同步、远程数据库故障恢复
和真实 MySQL/MariaDB/PostgreSQL 运行验证仍未完成。

## 10. 常见问题

| 现象 | 处理 |
|------|------|
| `/dungeon start` 提示人数不足/超限 | 调整队伍人数到模板 [最少,最多] 区间 |
| 开本失败（余额不足/网格已满/模板异常）| 查 Vault 余额、网格 freeCount、模板是否存在 |
| 建了实例但没传送 | 查门票状态、Schematic、`ks-InstanceWorld` 准备结果和数据库队列日志 |
| 改了 `grid.*` 没生效 | 网格项是「下次副本」级，已开副本不变 |
| Boss 死亡后没有通关 | 确认地图标记带 `/boss`、目标成功生成，且同一实例的其他登记 Boss 也已死亡 |
| 通关后奖励仍是 `PENDING` | 查 `ks_dungeon_reward_grants.last_error`；离线 proof、外部命令结果未知等必须人工核对 |
| 玩家离开后没有通关奖励 | 这是 `LEFT` 资格规则；通关快照只包含尚未离开的参与者 |
| 门票为 `REVIEW_REQUIRED` | Vault 外部调用结果不确定；先对账再人工处理，禁止直接重复扣款/退款 |
| 改 Java/资源后仍是旧的 | extra JAR 必须在 `plugins/ks-Eco/extra/` 且整服 restart |
