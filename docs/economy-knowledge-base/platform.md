# Economy Platform Contract

> [English](platform.en.md) | 中文

## 产品定位

`ks-Eco` 是可装配的服务器经济平台，不只是余额插件：

- `ks-core`：共享 SQLite、Web 网关、会话和跨插件桥接。
- `ks-Eco`：市场、价格、结算、完整物品存储、盲盒、限时销售、补偿、GUI、Web 和审计。
- Extra：银行、企业、税收、房地产、副本和政治等可选业务域。

核心经济必须在某个 Extra 缺失或启用失败时继续工作。Extra 只有在 `onEnable` 成功后才对 Web 和功能门控可见，禁用前先隐藏。

## 入口原则

- 游戏 GUI：高频、短路径、反复执行的玩家操作。
- 命令：余额、打开入口、快速查询或明确的管理动作。
- Web：长列表、地图、历史、批量配置和跨模块管理。
- 玩家 Web 不承担市场购买、挂单创建、求购创建、物流或暂存箱领取；这些保留在游戏 GUI。
- Web 当前是桌面优先、游戏命令登录的本地管理面板。除非出现明确需求，不做独立手机适配，也不把公开 HTTPS 场景当成既定前提。
- Web 详情采用“概览/地图/列表 -> 侧边详情与动作”；不制造假历史、假图表或占位业务数据。

## 数据与价值契约

- SQLite 由 `ks-core` 统一提供；模块使用自己的表前缀和明确的 Provider/共享表连接。
- 物品价值依赖完整 `ItemStack`：名称、Lore、附魔、属性、CustomModelData、PDC 和潜影盒内容都不能退化成 Material 名称。
- 背包溢出进入 `ks_eco_storage`；正常结算不能静默丢物品。
- 市场交易、官方收购、测试交易、补偿、盲盒和企业资金操作必须留下可解释记录。
- 数据库提交与外部 Vault 结算之间仍缺少统一的持久化 settlement journal，这是当前系统性风险。

## 线程契约

服务器线程专属：

- Bukkit/Paper 活对象、玩家、背包、GUI、世界、配方注册表。
- `ItemStack` 序列化/反序列化、PDC 和物品元数据。
- Vault 检查、扣款和退款。

工作线程职责：

- SQLite 读写和事务。
- 不可变 DTO、原始字节和纯计算。
- 排序、聚合、价格计算和 RNG。

标准流程：服务器线程快照 -> 工作线程预留/计算 -> 服务器线程应用 Vault/Bukkit 效果 -> 工作线程持久化结果或补偿。事务不得等待服务器线程回调。

## 权限契约

企业权限由企业自己的角色模板和个人授权组成，所有者拥有全部权限。当前关键权限包括：

- `MANAGE_FUNDS`：发薪、融资、还款、公户和其他资金管理。
- `BLINDBOX_DRAW`：代表企业执行一次盲盒抽取。
- `MANAGE_BIDDING`、`DECLARE_DIVIDEND`、`VIEW_FINANCE` 等保持各自职责。

银行同样使用角色模板和个人授权；发放贷款与审批贷款是两个独立权限。权限必须在服务器端执行，不能只靠页面隐藏按钮。

`BLINDBOX_DRAW` 不等于企业资金管理授权。任何批量预付、预算冻结或长期资金承诺都应要求 `MANAGE_FUNDS`。
