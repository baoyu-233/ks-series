# Enterprise Blind Box Ticket Audit

> [English](enterprise-blindbox-tickets.en.md) | 中文

审计日期：2026-07-16。范围是当前源码和本地测试服只读数据库，不代表生产服使用量。

## 结论

当前“企业购买式盲盒票券”没有形成独立玩法价值，已于 2026-07-16 退役。

它的唯一经济效果是：按当前单抽价从企业公户一次性预付若干次，之后使用时不再扣款。没有批量折扣、员工领取、指定持有人、预算审批、企业仓库、活动来源或额外奖励；购买者还可以自行设置到期时间，承担额外损失风险。

## 退役前实现（历史）

- 表：`ks_bb_tickets`、`ks_bb_ticket_log`。
- 购买：`BlindBoxManager.purchaseTicket`。
- 使用：`BlindBoxManager.pullWithTicket`、`pullForEnterpriseTicket`。
- Web：`handleEntBbTicketBuy`、`handleEntBbTicketsList`、`handleEntBbTicketUse`。
- 游戏 GUI：`EntBlindBoxGui` 只能列出和使用，不能购买。
- 权限：购买、查看和使用全部只检查 `BLINDBOX_DRAW`。
- 票券属于企业和奖池，不包含员工 UUID、角色、领取状态或分配记录。

页面宣称“可分发给员工”，但 schema 和 API 都没有分发动作。实际行为是所有拥有该企业 `BLINDBOX_DRAW` 权限的人共享同一批剩余次数。

## 关键问题

### P0：票券路径可绕过奖池边界

直接企业抽取会拒绝 `PLAYER` 池和 `limitedOnly` 池；票券购买只读取池价格，票券使用也没有复查 `ownerType` 或 `limitedOnly`。因此票券可成为绕过企业抽取限制的入口。

证据：

- `BlindBoxManager.purchaseTicket` 只调用 `getPool` 并计算价格。
- `pullForEnterprise` 检查 `ownerType` 和 `limitedOnly`。
- `pullForEnterpriseTicket` 只检查 enabled、行业、地块和分类。

### P0：失败返还和使用日志不是可靠事务

`pullWithTicket` 先提交扣除一张票，再进行物品抽取。随后成功日志或失败返还使用同一个 `autoCommit=false` 连接，但没有第二次 `commit`。连接关闭时这些写入不能被视为可靠提交。进程在扣票提交后、奖励发放前崩溃也会永久损失票券。

### P1：资金权限过宽

`BLINDBOX_DRAW` 的含义是执行一次抽取，但 Web 允许拥有该权限的员工一次性购买任意数量、任意有效期的票券并立即扣企业公户。长期或批量资金承诺应要求 `MANAGE_FUNDS`，并设置金额/数量上限。

### P1：线程边界不安全

- 游戏 GUI 在服务器线程同步执行多次 SQL、物品反序列化和抽取。
- Web 在 HTTP 线程调用同一同步方法，又会访问 Bukkit、`Player`、`ItemStack` 和背包。
- 票券尚未接入已经完成的玩家盲盒异步批量管线。

### P1：没有真正的“分发”

票券没有 assignee、领取、转移或角色额度。奖励直接进入操作者个人背包/暂存箱，不进入企业资产。企业承担成本，个人取得物品，但系统没有额外审批或归属记录。

### P2：没有购买理由

- 票价等于当前池价，无折扣。
- 直接公户抽取已经存在，保底同样按 `ent:<enterpriseId>` 共享推进。
- 票券增加过期、列表和管理成本，却没有新增能力。
- 可能锁定旧价格，但这没有被定义为正式产品能力，永久票券还会绕过未来调价。

### P2：界面和校验不完整

- 游戏 GUI 不能购买，Web 需要手填池 ID。
- 数量和有效天数没有合理上限；0 或负天数会生成永久票券。
- 列表仍展示已过期但有余额的票券，点击后才报错。
- Web 的 `enterpriseId` 参数没有与票券归属做显式一致性校验，虽然内部会再次检查实际企业权限。

## 推荐方案

### 方案 A：退役购买式票券（推荐）

1. 隐藏票券购买/使用 UI，拒绝新购买。
2. 保留表和只读查询，为可能存在的生产数据提供迁移窗口。
3. 若生产数据库同样无票券，后续从重置类别和文档中移除。
4. 企业盲盒只保留按次公户结算，并先完成企业抽取线程分离。

优点：最小复杂度，与当前零采用数据一致，直接消除绕过入口。

### 方案 B：改成企业抽奖预算

资金管理员创建预算：企业、奖池、总金额/次数、适用角色或员工、有效期和单人上限。员工只需 `BLINDBOX_DRAW`，每次抽取消耗预算。预算创建要求 `MANAGE_FUNDS`，所有分配和使用都有审计。

这比“共享纸票”更贴近企业治理，也不需要伪装成可分发物品。

### 方案 C：保留票券概念但改为奖励凭证

票券只能来自管理员、活动、合同或招标奖励，不再按原价由企业购买；必须包含 assignee/role、来源、领取状态和不可绕过的池门控。这样它才提供“延迟奖励/定向授权”这一独立价值。

## 决策状态

当前为 `ACCEPTED / RETIRED`：已采用方案 A。活跃代码不再创建、购买、列出或使用票券，Web 与游戏 GUI 入口均已删除，经济重置也不再操作遗留票券表。为避免破坏未知生产数据，现有表未执行 `DROP TABLE`；未来若重新引入授权玩法，必须按方案 B 或 C 重新设计，不能复活旧共享计数模型。
