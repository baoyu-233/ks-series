# 内容架构

> [English](content-architecture.md) | 中文

KS-Series 以基础生存为底，经济负责资源与货币结算，RPG 负责渐进成长和战斗证明，探索负责区域身份，
副本负责合作挑战，长期内容通过默认关闭的赛季基础逐步加入。第二/第三波的具体 ID、包状态和冲突见
`second-third-wave-content.zh-CN.md`。

## 持久生存层

原版生存、土地与房产、玩家交易、官方市场、仓储、企业和社区工程组成长期世界。住宅、机器、物流、
经济历史和社交关系不能因为新赛季或新 Boss 被清空。普通生存不要求职业、每日任务、Boss 通关或装备评分。

## 装备与战斗内容层

- MMOItems 负责物品 ID、静态属性、品质、展示和制作输入；它不单独授予账号证明。
- MMOInventory 只提供两枚戒指和一枚护符的小型装备面，不保存副本或赛季状态。
- ks-RPG 负责文件化武器/护符/戒指技能、区域掉落、材料兑换、内容物品 API、账号绑定证明和门槛。
- MythicMobs 内容包负责精英、Boss、helper、阶段、预警和成功/失败反馈；ModelEngine 与 ItemsAdder
  只负责可读视觉和资源。
- `ks-BossCombat` 当前仍保留 Frostbound 武器适应等窄范围跨插件规则；在迁移完成并实机验证前，
  不能假设它已经被 ks-RPG 完全替代。

第二波灰烬/风暴内容已经进入 ks-RPG source，并有独立 MMOItems/MythicMobs 数据包；第三波仍只存在于
独立 staging 包。包内可解析不等于生产服已经拥有对应物品、模型、区域或奖励。

## 副本与合作层

`ks-Eco-RealEstateDungeon` 负责模板、队伍名单、门票、复活、实例生命周期和完成奖励；
`ks-InstanceWorld` 负责实例世界与网格。副本奖励应以权威参与名单为准，不由 Boss 死亡半径或最后一击决定。

可购买内容限定为门票、补给、修理、复活、基础材料和基础装备。关键主动能力与遗物突破必须满足：

`可交易材料 + CASH/制作消耗 + 账号绑定战斗证明`

当前 proof gate 只能检查，尚不能原子扣除材料与货币；副本模板中的 `entry-gate` 也只是设计元数据。
因此第三波遗物件必须保持不可获取，直到结算和技能门槛同时接线。

## 区域与长期追逐

区域通过特有材料、低频精英、委托和制作关系形成身份，不通过无限 RPG 经验或强制每日签到形成压力。
组队可以提高效率和解法数量，基础生存与基础配装仍允许单人稳步前进。长期 `triune_atlas` 类目标只适合
开放横向配方、技能视觉、称号、房屋奖杯和社区工程，不应无限叠加常驻伤害。

## 赛季基础层

ks-RPG 已实现本地 SQLite 赛季领域与存储基础：赛季生命周期、区域声望周上限与追赶额度、幂等事件快照、
社区项目、奖励领取状态和归档记录。数据库操作由单工作线程执行，服务器线程只读取缓存状态。

该系统默认 `season.enabled=false`。显式开启目前只初始化空 `season.db`，不会自动创建赛季、启动事件、
授予声望或修改玩家。正式内容目录、调度、奖励执行、管理员流程、远程数据库和跨服协调仍未接线。

## 所有权与线程边界

- Bukkit/Paper 实体、玩家、物品栏、ItemStack 元数据、GUI、MythicMobs 和 Vault 调用留在服务器线程。
- SQL 与纯计算进入工作线程，只传递不可变快照和结果，不跨服务器线程回调持有事务。
- 奖励、证明和赛季领取必须幂等；失败重试不能重复发钱、发物品或开放突破。
- 知识库记录产品契约和验证状态；最终行为仍以当前代码、配置和维护窗口实测为准。
