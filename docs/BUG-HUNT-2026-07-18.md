## P0 - 严重

### 副本复活可对无效实例和存活玩家重复扣款

- 入口：`ks-Eco-RealEstateDungeon/DungeonWebHandler.java:261-280`
- 结算：`DungeonDeathHandler.java:77-93`
- 持久化：`DungeonInstanceManager.java:923-950`
- 未验证实例属于当前玩家、实例仍 ACTIVE、参与者为 DEAD；participant UPDATE 影响 0 行也返回成功。
- 不存在的 instanceId、重复请求、数据库失败均可能真实扣款并显示复活成功。

### 副本先提交 COMPLETED，后发奖励，崩溃后永久漏发

- `DungeonLifecycleStore.java:42-70` 先提交终态、LEFT 和清理。
- `DungeonInstanceManager.java:697-706` 提交后才调用奖励。
- 重试时 `transitioned=false`，奖励名单为空；第一个玩家奖励异常还会阻断后续玩家。

### 企业分红跨服启动存在双付竞态

- `EnterpriseManager.java:29` 每个节点启动时都执行恢复。
- `DividendSettlementJournal.java:110-123` 无 owner/server/lease 条件地把共享库全部 PENDING 改成
  COMPENSATION_REQUIRED。
- B 节点可在 A 节点实际 Vault 入账前抢先改状态；A 随后无法 markPaid，人工补偿可能再次付款。

## P1 - 高

### 跨服轮询会漏事件或只让一台服务器消费广播

- `JdbcDatabaseTransportStore.java:289` 使用业务时间复合游标，晚提交的较早事件会永久落在游标后方。
- `JdbcDatabaseTransportStore.java:257` 游标主键只有 consumer_id，多服同名 consumer 会共享并互相推进游标。
- `DatabasePollingTransport.java:251` CAS 失败后不重载数据库游标，会永久重放并永久 CAS 失败。

### 跨服事件的幂等重复发布会误报冲突

- `JdbcDatabaseTransportStore.java:379` 使用 record equals 比较事件。
- `TransportEvent.java:18` 含 byte[]，Java record 默认按数组引用比较；数据库读回的新数组必然不相等。

### 跨服缓存失效可永久陈旧

- `CrossServerCacheInvalidationCoordinator.java:224` 把节点 HLC 当全局提交版本；慢时钟节点后提交的唯一事件会被判 STALE。
- 监听器失败前已经写入去重和版本，重投只能得到 DUPLICATE，无法重试。
- future HLC 还会被当成 transport availableAt，时钟偏移可让失效事件延迟数小时。

### MySQL/PostgreSQL 并不能运行当前 ks-Eco 业务 schema

- `ListingManager.java:29` 使用 MySQL 不允许的 TEXT 主键和 PostgreSQL 不存在的 BLOB。
- `LimitedSaleManager.java:153-186` 仍含 BLOB/AUTOINCREMENT。
- `KsEco.java`、bank、enterprise、politic 等仍有大量 INSERT OR IGNORE/REPLACE、strftime 和 SQLite DDL。
- `ks-core` 连接层支持远程数据库，但不能据此宣称整个 ks-Eco 已支持远程数据库。

### 限时售卖跨服可超卖，失败退款可能退到错误后端

- `LimitedSaleManager.java:772-790` 读取后更新只按 id，不校验旧库存、价格或 currency_id，也没有行锁。
- 两服可同时购买最后一件并都成功。
- `LimitedSaleManager.java:1030-1045` 扣款和退款重新选择 Vault/Builtin；Vault 状态变化时会退到另一个余额源，且 deposit=false 仍显示已退款。

### 企业 Vault 超时会出现“接口失败但稍后扣款”

- `ServerThreadVaultGateway.java:29-104` Future 超时后不取消排队任务。
- 主线程恢复后 withdraw 仍会执行，但业务已经按失败返回，注资/注册可能只扣款不入账。

### 企业分红 PARTIAL 重试会重复支付已成功收款人

- `EnterpriseManager.java:956-1003` 部分成员成功、部分失败时返回失败并标 PARTIAL。
- PARTIAL 不阻断新批次；用户重试会再次支付上一批已成功的成员。

### 银行贷款 ACTIVE 提交后才发本金

- `BankManager.java:1393-1421` 先提交贷款、申请状态和资产扣减，再调用 Vault 入账。
- 提交后崩溃会留下 ACTIVE 贷款和已减少银行资产，但玩家没有收到本金，且无恢复 journal。

### 税款扣除后断库可绕过退款

- `TaxManager.java:141-180` 只捕获 SQLException；`KsDataStore.getConnection()` 断连会抛 IllegalStateException。
- 异步任务直接退出，无税务记录，也不会调度退款。

### 地产缓存失败或 reload 会造成保护绕过

- `RealEstateManager.java:119-158` 刷新失败后仍发布空/部分快照，保护查询随后 fail-open。
- Extra reload 前排队的 revoke 只更新旧 manager 缓存，新保护 listener 可永久保留已撤销权限。
- 跨服没有任何地产边界/信任缓存失效接线。

### HWP 私有备注搜索泄露

- `HwpWebHandler.java:358-375` 允许无 token 搜索。
- `MapAnnotationManager.java:158-185` 没有 public/owner 条件，返回所有匹配备注及详情。

### HWP debug API 未鉴权且会执行昂贵渲染

- `HwpWebHandler.java:111-116` 未把 `/api/debug` 列为管理员路由。
- `HwpWebHandler.java:620-675` 即使无 token 仍收集 Bukkit 状态，并阻塞等待一次 tile render。

### Maven 测试是假绿

- `ks-Eco/pom.xml` 没有 JUnit/TestNG。
- 当前 5 个 `*Test` 只有 main，无 `@Test`；Surefire 实际执行 0 tests。
- `mvn test/package` 成功只证明测试源码可编译，不能证明跨服或多货币行为通过测试。

## 第二批已确认 P1

### InstanceWorld 延迟释放会释放其他实例正在使用的网格

- `InstanceStore.java:254-280` 只按 grid_id 标记 FREE，不校验 occupied_by=instanceId。
- A 释放、B 复用同一网格后，迟到的第二次 finishRelease(A) 会把 B 的活跃网格释放。

### InstanceWorld 清理范围小于 schematic 实际边界

- `CanvasService.java:24-43` 可粘贴超过 arena radius 的完整 schematic。
- `InstanceWorldService.java:391-417` 释放仍只清配置半径和固定 Y 范围，外围方块会永久残留。

### ItemEditor 可永久丢失精炼物品

- `RefineCommand.java:57-58` 无已有会话检查，第二次 /refine 直接覆盖第一件物品的会话。
- 退出或取消只归还第二件，第一件不再被任何 map 引用。

### ItemEditor 模板加载会销毁完整物品数据

- `DesignCommand.java:105-124` 和 `EditSession.java:202-204` 重新创建并覆盖 ItemStack。
- 数量、耐久、容器内容、第三方 PDC/MMOItems 标识会永久丢失。

### Inherit 发物和 DELIVERED 状态不原子，可重复领取

- `KsInherit.java:465-498` 先把物品放入背包，最后才更新状态。
- 状态更新失败后重试会再次发放同一物品。

### RPG proof gate 的错误 YAML 类型仍会 fail-open

- `ProgressionCatalog.java:68` 对标量 required-proofs 调 getStringList，得到空列表。
- `ProgressionService.java:103-106` 将空 requirements 判定为已满足。

### RPG 错误 config.yml 会替换当前有效 catalog

- `KsRpg.java:51` 使用 reloadConfig；Paper 的 YAML loader 记录语法异常后可返回空/默认配置。
- 后续仍交换 catalog 并报告成功，没有保留旧运行态。

### 政治表决在剩余票仍可逆转结果时提前结束

- `VoteManager.java:182` 达到法定人数后立即按当前多数结束。
- 8 席中前 5 票为 3 YES/2 NO 会提前通过，即使剩余 3 票全部 NO。

### HWP clear-cache 可递归删除 tiles 根目录外内容

- `TileStore.java:71-76` 对 world 执行 baseDir.resolve(world) 后直接 Files.walk 删除。
- 缺少 normalize/startsWith；管理员 token 请求 world=../.. 可把删除范围扩大到插件目录。

### Title IA 上传可触发图片解压炸弹

- `TitleWebHandler.java:415,475,574-593` 只限制压缩请求体，不限制解码后的图片尺寸。
- 高压缩 8192x8192 PNG 可分配数百 MiB 并逐像素扫描。

### Sentinel 数据库短暂失败会永久丢失整批审计

- `KsSentinel.java:239` 写库前先 poll 清空队列。
- `KsSentinel.java:270-273` 失败只 rollback，不把记录重新入队。

### KSBot 可绕过禁止无限动作

- `BotManagerModule.java:831` 只在用户输入 times=-1 时检查 actionAllowInfinite。
- Leaves 的 sneak/swim/move 状态动作内部固定无限次数，计数 setter 不存在又被静默忽略。

### 多货币幂等键在 MySQL 默认排序规则下会碰撞

- `JdbcCurrencyLedger.java:50` 使用普通 VARCHAR 主键，IdempotencyKey 保留大小写。
- 默认大小写不敏感 collation 下 BUY:A 与 buy:a 会被视为同一 operation，第二笔可能免扣款成功重放。

### 多货币手续费可通过拆单绕过

- `ExchangeRule.java:53` 按目标货币普通舍入规则计算手续费和净额。
- 整数货币 49% 手续费下，100 次兑换 1 单位可全部免手续费，而一次兑换 100 会收 49。

### 跨服租约可被迟到事务复活

- `JdbcDistributedLeaseLock.java:136` 先读取数据库时间，再用该旧时间续租。
- 另一事务释放后，旧事务仍可能满足 expires_at > oldNow 并重新续租、执行 fenced 回调。

### BotGuard 在 MMO 插件内部 reload 后可能恢复重复监听器

- `KsBotGuard.java:178,287` 保存旧 original，但恢复时不确认 wrapper 是否仍注册，也不识别插件已注册的新 listener。
- BotGuard 停用后可能形成 old original + new listener 双触发。

### 批次 5：剩余缺陷与测试可信度

- 多货币标识列在 MySQL/MariaDB 使用 `utf8mb4_bin`，并为已有表补迁移，大小写不同的幂等键不再因默认排序规则碰撞。
- 兑换手续费按目标最小单位向上取整，关闭小额拆单把手续费舍入为零的路径；alternative payment 必须有至少两种不同币种，未知/不可直接支付币种、posting 不守恒和同幂等键换币种均 fail closed。
- BotGuard 增加主线程周期 reconcile、original/wrapper 身份去重和失效追踪清理；只有仍实际注册的 wrapper 才允许在 BotGuard 停用时恢复 original，MMO 内部 reload 不再复活旧 listener。
- ks-Eco 的原 main 合同套件已接入 JUnit/Surefire，`mvn test` 不再以“只编译、零执行”冒充通过。该关闭项只代表本地合同测试真实执行，不代表远程数据库或 Paper 实机已通过。

### 批次 6：经济、地产与副本可靠性

- 银行放款采用 `PENDING_PAYOUT -> PAYOUT_SETTLING -> ACTIVE`；重启遇到外部入账结果不确定时进入 `RECONCILE_REQUIRED`，不再删除贷款并返还银行本金。三条放款路径均接入该状态机。
- 银行存取款和周期任务按旧余额先结息，结息状态、唯一 posting、账户余额和银行流动性在同一事务提交；新增 300-850 信用评分、A-E 等级、动态额度/期限、风险与期限加点及固定报价确认基础。

## 2026-07-19 基础设施、P0/P1 与经济收口增量
